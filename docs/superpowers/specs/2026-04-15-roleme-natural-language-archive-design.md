# roleMe 自然语言归档设计

日期：2026-04-15

## 背景

当前 `roleMe` 已经具备以下能力：

- resident / on-demand 角色上下文加载
- `USER.md`、`MEMORY.md`、`episodes/`、`brain/`、`projects/` 的结构边界
- workflow 归档的确定性写入
- 底层记忆写入工具

但日常使用体验仍然偏“显式命令式”：

- 用户需要主动说“记一下”“总结并归档”
- 用户经常需要自己想“这条该放 USER 还是 MEMORY”
- 对于学习沉淀、项目复盘、方法论提炼这类明显值得保存的内容，skill 还没有形成统一的默认行为约定

这会让 `roleMe` 更像一个需要手工操作的存储结构，而不是一个自然语言驱动的归档助手。

## 目标

在**不增加新代码**的前提下，仅通过 `skill/SKILL.md` 与 `skill/references/usage.md` 的文档约定，建立一套更顺手的默认归档行为：

- 用户只需要自然表达内容，不必每次手动指定归档位置
- 当角色已加载后，模型对明显值得保存的内容应主动归档
- 归档前先做提炼，再决定落点
- 不确定时默认保守落盘，不轻易污染 resident 层
- 归档完成后给用户一个极短回执

## 非目标

- 不新增统一的 memory router 代码入口
- 不修改 `tools/memory.py` 的职责边界
- 不实现新的自动分类 runtime
- 不在本次设计中处理复杂的多目标拆分推理优化

## 设计原则

### 1. 用户负责表达，模型负责归档

用户只需要自然表达正在发生的学习、总结、复盘、偏好确认与方法论提炼，不需要每次指定：

- 存到哪里
- 是否要压缩
- 是 `USER`、`MEMORY`、`episode` 还是 `brain`

模型负责：

- 识别可归档信号
- 提炼原始内容
- 判断归档位置
- 执行归档
- 告知结果

### 2. 先提炼，再落盘

默认不应把原始聊天内容整段原样写入 resident 层。

归档前应先压缩成：

- 稳定偏好
- 长期结论
- 项目经验
- 主题知识
- 过程记录

然后再决定归档位置。

### 3. 高价值主动归档，低置信度保守处理

当内容明显属于以下类型时，模型应主动归档，而不必等待用户显式下命令：

- 学习沉淀
- 项目复盘
- 方法论提炼
- 稳定偏好
- 长期协作规则

当判断不够确定时：

- 优先落到 `memory/episodes/`
- 或优先落到项目级记忆
- 不轻易提升到 `memory/USER.md` / `memory/MEMORY.md`

### 4. resident 层宁少勿噪

`USER.md` 与 `MEMORY.md` 是 resident 层，会影响后续会话的默认理解。

因此应采用保守准入策略：

- `USER.md` 只接收稳定偏好、长期约定、持续成立的事实
- `MEMORY.md` 只接收高价值长期结论、方法论摘要、可复用认知

## 默认触发场景

当角色已加载后，出现以下表达时，模型应默认进入“提炼 + 归档”流程：

- “我今天学到一个点……”
- “这篇文章的核心对我很重要……”
- “这个以后可以作为通用工作方式……”
- “这次项目踩坑值得记一下……”
- “以后默认这样和我协作……”
- “这条是长期有效的规则……”

同时，不要求用户必须使用“归档”“记忆”“保存”这类显式词。

## 归档判断顺序

默认按以下顺序判断：

1. 是否属于项目上下文  
   如果明显绑定某个项目、仓库、迭代或需求，优先落到 `projects/<project-slug>/...`

2. 是否属于稳定偏好或长期协作约定  
   如果明显在表达“以后默认这样”“这是我的固定偏好/规则”，写入 `memory/USER.md`

3. 是否属于长期有效的结论或方法论  
   如果是从学习、复盘、实践中提炼出的可复用结论，写入 `memory/MEMORY.md`

4. 是否属于主题化知识  
   如果已经形成可反复引用的专题知识，整理到 `brain/topics/*.md` 并更新 `brain/index.md`

5. 其他情况  
   默认写入 `memory/episodes/`

## 默认保守策略

### 项目优先于全局

同一句内容若既有项目经验又有通用结论，优先先落项目级位置。

只有在模型高置信度判断其可跨项目复用时，才额外提升到全局 `MEMORY.md`。

### USER 最严格

不要把一次性要求、当前任务指令、临时偏好写入 `USER.md`。

### 不确定先记过程

模糊时先写 `episode`，不要抢先升格为 resident 规则。

### 回执短，不打断节奏

归档完成后，仅用一句极短回执说明结果，不进行长篇解释。

## 默认回执风格

推荐固定使用短句：

- 已归档到项目记忆。
- 已提炼为长期结论，写入 MEMORY。
- 已记为稳定偏好，写入 USER。
- 已先记录为 episode。
- 已整理为 brain 主题，并加入知识索引。

如果涉及多层归档：

- 已归档：过程写入 episode，长期结论写入 MEMORY。
- 已归档：项目经验写入项目记忆，通用规则提升到 USER。

如果本次更新 resident 层：

- 这次更新了 resident 记忆，重新执行 `/roleMe <角色名>` 后会刷新当前会话底座。

## 文档落点

### `skill/SKILL.md`

只增加高层运行时规则，保持短小：

- 角色已加载后，对明显值得保存的内容主动归档
- 归档前先提炼，再决定归档位置
- 低置信度优先落 `episode` 或项目记忆
- 更新 resident 后提醒 reload

### `skill/references/usage.md`

新增一节“自然语言归档”，详细说明：

- 用户无需指定路径
- 默认触发场景
- 归档判断顺序
- 低置信度处理
- 回执风格

## 风险与取舍

### 风险

- 文档约定增强后，不同模型对“值得保存”的判断仍可能存在轻微波动
- 由于本次不改 runtime，自动归档能力仍依赖 skill 执行质量
- 若文档写得过长，会增加 skill 运行时上下文负担

### 取舍

本次优先解决“使用约定不一致”的问题，而不是一次性把自然语言归档产品化。

理由：

- 当前仓库已经具备基础结构与写入工具
- 真正缺的是一套稳定的一致性行为约定
- 先通过文档层跑通使用模式，再决定是否值得追加统一 router 代码，更稳

## 预期结果

本次优化完成后，`roleMe` 的使用体验应更接近：

- 用户自然表达内容
- 模型自动提炼
- 模型自动分类归档
- 用户只收到极短结果回执

而不是：

- 用户先判断该存哪
- 用户手工下多步命令
- 模型只做被动写文件
