# 可吸收点：把工作方式沉淀成 Workflow Memory

这份文档不再讨论 `roleMe` 和其他工作空间的整体对比，只收敛一个最值得吸收并改造进 `roleMe` 的点：

> 不要复制整个 AI workspace，而是把“工作方式”提炼成可检索、可路由、可增长的 `workflow memory`。

这不是一个附属功能，而是 `roleMe` 从“用户画像 + 长期记忆”继续往前走时，最有机会形成产品壁垒的一步。

## 一句话结论

`roleMe` 真正值得吸收的，不是完整的工作流系统，而是：

> 让系统不只记住“用户是谁”，还记住“用户通常怎么做事”。

## 为什么这是精华

如果只做 persona 和 memory，`roleMe` 会更像一个高质量的用户记忆系统：

- 它知道用户是谁
- 它知道用户偏好什么
- 它知道用户过去说过什么

但它未必知道：

- 遇到需求分析时，该按什么步骤展开
- 遇到产品讨论时，该如何组织信息
- 遇到项目协作时，该优先调哪种工作方式

这就是 `workflow memory` 的价值。

它让系统开始记住“协作方法”本身，而不仅仅是“协作对象”。

## 应该吸收什么

真正值得吸收的只有下面几件事。

### 1. 把 workflow 当成记忆对象，而不是任务引擎

不要先把它理解成一个庞大的流程编排系统。

在 `roleMe` 里，更合适的定义是：

- workflow 是一种长期有效的方法记忆
- workflow 是可以被 query 命中的上下文单元
- workflow 是“该怎么做”的结构化答案，而不是“现在就开始执行”的强制控制器

换句话说，workflow 在这里首先是 memory，其次才可能是 orchestration。

### 2. 一个 workflow，一个文件

不要做成一个巨大的总说明，也不要把所有方法论堆进 `AGENT.md`。

更合适的组织方式是：

- 一个 workflow 一个文件
- 用 index 做摘要入口
- 只在 query 命中时注入对应 workflow

这会带来三个直接好处：

- 方法边界清晰
- 便于增量沉淀
- 不会把常驻上下文越堆越大

### 3. 保留 project / global 双层结构

workflow 不应该只有一层。

更合理的是：

- `brain/workflows/` 放通用工作方式
- `projects/<project>/workflows/` 放项目特有工作方式

这样用户既可以拥有稳定的通用协作方式，也可以在某个项目里逐步长出更具体的项目工作法。

### 4. 让 workflow 进入上下文路由，而不是常驻加载

workflow 最大的价值，不是在启动时全部加载，而是在需要时被正确发现。

所以它应该进入 `context_router` 的命中链路：

- 先判断 query 是否命中 project workflow
- 未命中时再回退到 global workflow
- 命中置信度不足时宁可不注入，也不要误注入

这保证了 workflow memory 是“准确的增强”，而不是“持续的噪音”。

## 不应该吸收什么

如果不做裁剪，`roleMe` 很容易被拖向另一个重型 workspace。下面这些都不该先吸收。

### 1. 不要先吸收完整的模式系统

例如：

- 讨论模式
- 汇报模式
- 产品设计模式
- 临时任务模式

这些适合工作空间产品，不适合当前阶段的 `roleMe`。

`roleMe` 应该先记住方法，而不是先控制所有工作模式。

### 2. 不要先吸收重型资料库

大型资产库、知识库、汇报材料库都很有价值，但它们不属于这个产品的最小核心。

`roleMe` 的第一性问题不是“存更多资料”，而是“在正确的时候调出正确的上下文”。

### 3. 不要让 workflow 反过来污染 resident 层

resident 层应该保持小而稳。

workflow 应该以索引摘要的方式进入 resident snapshot，必要时再展开具体文档，而不是整篇常驻。

## 这件事对 roleMe 的产品意义

如果把这个点做好，`roleMe` 的产品表达会发生变化。

之前更像：

> 一个能长期记住用户身份和偏好的角色系统

做完之后更像：

> 一个能长期记住用户身份、上下文和工作方式的协作运行时

这会让 `roleMe` 和普通 AI 记忆产品产生更清晰的差异：

- 普通产品主要记“用户是谁”
- `roleMe` 开始记“用户怎么工作”

后者更接近长期协作的真实价值。

## 对 roleMe 的落地建议

这一点不需要大改架构，反而应该继续保持轻量。

建议按下面的方式理解和推进：

### 当前定义

把 `workflow memory` 定义为：

- 一种按需注入的协作方法记忆
- 一种可被 query 命中的上下文资产
- 一种 project/global 双层组织的长期结构

### 当前边界

它不负责：

- 全流程任务编排
- 多模式会话控制
- 完整工作空间管理

它只负责：

- 存方法
- 找方法
- 注入方法

### 当前正确的实现方向

- 用 `workflow index` 做入口摘要
- 用单文件 workflow 承载具体方法
- 用 `context_router` 负责命中与注入
- 用归档能力把稳定工作方式写回 workflow

## 建议写进产品定义的一句话

如果要把这个点沉淀成后续产品定义，可以直接写成：

> `roleMe` 不只是记住用户身份，还会把稳定的工作方式沉淀为可按需调用的 workflow memory。

## 最后的判断

如果只能从外部工作空间里吸收一个点，我会优先吸收这个：

> 把“工作方式”从零散经验，升级为 `roleMe` 内部的一等上下文对象。

因为这一步最轻，却最容易把产品从“记忆系统”推向“协作运行时”。
